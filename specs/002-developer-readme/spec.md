# Feature Specification: Developer Onboarding & README

**Feature Branch**: `002-developer-readme`  
**Created**: 2026-04-20  
**Status**: Draft  
**Input**: User description: "Our goal is to make repository and the generated binary/tool to be an inviting place for other developers to onboard and try the tool out. Let us add create a readme with a cool logo and a README that invites users. The primary utility of this tool is to allow scaling to multiple claude sessions - where the session is a theme you have decided as opposed to claude. Allows focus on the activity at hand."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — First-Time Developer Discovery (Priority: P1)

A developer discovers `cs` via GitHub or word of mouth. They land on the repository and within 60 seconds understand what the tool does, why they'd want it, and how to install it. The README communicates the core value proposition — you define the theme/focus of a session, not Claude — with enough personality that they want to try it.

**Why this priority**: First impressions determine whether someone installs the tool or moves on. A compelling README is the single highest-leverage onboarding artifact.

**Independent Test**: Share the README with a developer unfamiliar with the project. Ask them to explain back what `cs` does and whether they'd try it. Success if they nail the concept and express interest.

**Acceptance Scenarios**:

1. **Given** a developer visits the repository root, **When** they read the README, **Then** they can articulate that `cs` lets them run multiple focused Claude sessions — each defined by a human-chosen theme — without needing to know tmux.
2. **Given** a developer wants to try the tool, **When** they follow the README install section, **Then** they can have `cs` running in under 5 minutes on a supported platform.
3. **Given** a developer sees the logo/header, **When** they scan the page, **Then** the visual identity communicates "polished developer tool" and invites exploration.

---

### User Story 2 — Quick Install & First Run (Priority: P2)

A developer who has decided to try `cs` wants to go from zero to their first session as quickly as possible. The README provides a quick-start section that gets them to a working `cs` invocation with minimal ceremony.

**Why this priority**: Reducing install friction directly converts interest into active users.

**Independent Test**: Follow only the quick-start section from a clean machine. Measure time-to-first-session. Should be under 5 minutes on macOS with Homebrew or a direct binary download.

**Acceptance Scenarios**:

1. **Given** a macOS developer with standard tooling, **When** they follow the quick-start instructions, **Then** they run `cs` successfully for the first time within 5 minutes.
2. **Given** a developer who prefers building from source, **When** they follow the build instructions, **Then** they produce a working binary with a single command.
3. **Given** a developer who runs `cs` for the first time, **When** `cs setup` has not been run, **Then** the tool guides them to run setup rather than failing silently.

---

### User Story 3 — Understanding the Mental Model (Priority: P3)

A developer who has installed the tool wants to understand the philosophy behind session naming — that the session theme is chosen by them as a human, not auto-generated by Claude. The README conveys this clearly through framing and examples so users adopt the right mental model from day one.

**Why this priority**: Correct mental model adoption prevents confusion and increases long-term retention.

**Independent Test**: After reading the README (no other context), ask a developer: "Who names the sessions in cs?" They should answer "me/the developer" not "Claude."

**Acceptance Scenarios**:

1. **Given** a developer reads the README, **When** they reach the "how it works" or concept section, **Then** they understand that session names represent human-chosen themes (e.g., "auth-refactor", "blog-post", "api-design").
2. **Given** a developer reads the examples, **When** they see concrete session names, **Then** the examples reinforce focus and intentionality as the core benefit.

---

### Edge Cases

- What happens when a developer is on Linux or Windows? The README should clearly state supported platforms (macOS initially) and set expectations.
- What if a developer has an existing tmux setup? The README should clarify that `cs` uses its own isolated socket and will not interfere with existing tmux sessions.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The repository MUST have a `README.md` at the root that is the primary onboarding document for new developers.
- **FR-002**: The README MUST include a visual logo or ASCII art header that gives the project a distinct identity.
- **FR-003**: The README MUST open with a one-paragraph value proposition that explains what `cs` does and why it matters — focused on the human-session-ownership concept.
- **FR-004**: The README MUST include a quick-start section covering: installation (binary download or build from source), `cs setup`, and launching a first session.
- **FR-005**: The README MUST include a "how it works" or concept section that explains the mental model: the developer names sessions by theme, multiple parallel sessions are supported, and no tmux knowledge is required.
- **FR-006**: The README MUST list supported platforms and prerequisites (tmux, macOS).
- **FR-007**: The README MUST include a usage reference covering the primary commands (`cs`, `cs setup`, `cs list`, `cs --version`).
- **FR-008**: The README MUST include at least three concrete session name examples that illustrate focused, theme-based work (e.g., "refactor-auth", "write-changelog", "debug-api").
- **FR-009**: The README SHOULD include a brief section on contributing or linking to contribution guidelines, even if minimal.
- **FR-010**: The logo MUST be usable in GitHub's light and dark rendering modes without becoming illegible.

### Key Entities

- **README.md**: Root-level onboarding document; single source of truth for new users.
- **Logo/Visual Identity**: SVG, PNG, or ASCII art header; must render well on GitHub.
- **Quick-start section**: Numbered steps from install to first session.
- **Session examples**: Concrete named sessions that demonstrate the mental model.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A developer unfamiliar with the project can explain the core value proposition (human-chosen session themes, multi-session scaling, no tmux knowledge) after reading the README alone — validated by a brief comprehension check.
- **SC-002**: A developer on macOS can complete install and run their first `cs` session in under 5 minutes following only the quick-start section.
- **SC-003**: The README covers all primary commands such that a new user does not need to read source code or run `--help` to understand basic usage.
- **SC-004**: The visual identity (logo/header) is legible and renders correctly in both GitHub light and dark modes.
- **SC-005**: Zero support questions about "what does cs do" or "how do I install it" from developers who read the README — measured informally over the first month of public use.

## Assumptions

- The primary target platform for the quick-start is macOS; Linux instructions are a stretch goal and not required for this feature.
- The logo will be generated as an SVG or high-quality PNG; hand-crafted ASCII art is an acceptable fallback if tooling for image generation is unavailable.
- The README will live at the repository root (`README.md`) and be the only top-level onboarding document; a separate `CONTRIBUTING.md` is out of scope for this feature.
- The tool's name is `cs` (the binary) and the project/repository name is `claude-sessions`; both names should appear in the README.
- Users are expected to have Homebrew or Go installed for installation; no custom installer is in scope for this feature.
- The README does not need to document internal architecture or the tmux socket implementation — that belongs in developer documentation, not onboarding.
